# HW08-09 - PyTorch MLP: регуляризация и оптимизация обучения

## 1. Кратко: что сделано

- Выбран датасет `EMNIST(split="balanced")`: он заметно сложнее обычного MNIST из-за 47 классов, но при этом остаётся подходящим для честного сравнения MLP-конфигураций из S08-S09.
- В части A проведены четыре эксперимента: базовый `E1`, вариант с `Dropout` (`E2`), вариант с `BatchNorm` (`E3`) и дообучение лучшей конфигурации с `EarlyStopping` (`E4`).
- В части B на той же архитектуре выполнены три обязательных прогона: `O1` с слишком большим `lr`, `O2` с слишком маленьким `lr` и `O3` с `SGD+momentum` и `weight_decay`.

## 2. Среда и воспроизводимость

- Python: `3.13.5`
- torch / torchvision: `2.10.0` / `0.25.0`
- Устройство (CPU/GPU): канонический прогон зафиксирован на `cpu` для воспроизводимости; при необходимости можно переопределить через `HW0809_DEVICE=auto/cuda/mps`
- Seed: `42`
- Как запустить: из корня репозитория открыть `homeworks/HW08-09/HW08-09.ipynb` и выполнить `Run All`; по умолчанию ноутбук использует `cpu`, а путь к уже скачанному EMNIST при необходимости задаётся через `HW0809_DATA_ROOT`

## 3. Данные

- Датасет: `EMNIST(split="balanced")`
- Разделение: `train/val/test = 90240 / 22560 / 18800`
- Трансформации (transform): `ToTensor()` + `Normalize((0.5,), (0.5,))`
- Комментарий (1-3 предложения): в датасете 47 классов рукописных символов, поэтому для MLP важны стабильная оптимизация и аккуратная регуляризация. Размер входа небольшой (`1 x 28 x 28`), что позволяет честно сравнивать именно настройки обучения, а не сложные архитектуры.

## 4. Базовая модель и обучение

- Модель MLP (кратко): `Flatten -> Linear(784, 512) -> ReLU -> Linear(512, 256) -> ReLU -> Linear(256, 47)`
- Loss: `CrossEntropyLoss`
- Базовый Optimizer (для части A): `Adam(lr=1e-3)`
- Batch size: `128`
- Epochs (макс): `10` для `E1-E3`, `15` для `E4`, `6` для `O1-O2`, `10` для `O3`
- EarlyStopping: `patience=3`, метрика `val_accuracy`

## 5. Часть A (S08): регуляризация (E1-E4)

- E1 (base): 2 скрытых слоя `[512, 256]`, без `Dropout` и без `BatchNorm`
- E2 (Dropout): как `E1`, но с `Dropout(p=0.3)` после скрытых слоёв
- E3 (BatchNorm): как `E1`, но с `BatchNorm1d` после каждого `Linear`
- E4 (EarlyStopping): лучшая конфигурация из (`E2`, `E3`) + `EarlyStopping`

## 6. Часть B (S09): LR, оптимизаторы, weight decay (O1-O3)

- O1: слишком большой `lr` -> `Adam(lr=1.0)`
- O2: слишком маленький `lr` -> `Adam(lr=1e-5)`
- O3: `SGD(lr=5e-2, momentum=0.9, weight_decay=1e-4)`

## 8. Анализ

На `EMNIST Balanced` даже базовый MLP обучается вполне уверенно, но различия между конфигурациями становятся видны по validation-метрикам. В `E1` качество уже приемлемое, однако модель без дополнительной регуляризации уступает варианту с нормализацией. `Dropout` в `E2` не дал выигрыша: train/val разрыв становится чуть мягче, но сама сходимость получилась более консервативной, и итоговый `val_accuracy` оказался ниже базового варианта. `BatchNorm` в `E3`, наоборот, помог стабилизировать обучение и дал лучший результат среди кандидатов части A до ранней остановки.

В `E4` та же batchnorm-конфигурация была дообучена с `EarlyStopping`. Ранняя остановка сработала после 15-й эпохи, но лучшими остались веса с 12-й эпохи, поэтому в итоговый чекпойнт не попали более поздние и чуть менее удачные состояния модели. Это дало небольшой, но честный прирост по `val_accuracy` относительно `E3`, после чего модель была один раз оценена на test и показала `0.8521`.

Часть `HW09` хорошо видна именно на кривых обучения. В `O1` слишком большой learning rate почти срывает обучение: loss остаётся около `4`, а accuracy держится возле `2%`, то есть модель фактически не выходит из режима случайного угадывания. В `O2` learning rate слишком мал, поэтому за 6 эпох модель просто не успевает сдвинуться к хорошему решению. `O3` показывает более инженерный сценарий: `SGD+momentum` с `weight_decay` обучается стабильно и даёт сильный результат, особенно по `val_loss`, но без отдельного протокола выбора не должен подменять финальную модель из части A. Для этого датасета разумным итогом выглядит MLP `[512, 256]` с `BatchNorm`, `Adam(lr=1e-3)` и `EarlyStopping`.

## 9. Итоговый вывод

В качестве базового решения для этой домашней работы я бы оставил конфиг `E4`: MLP `[512, 256]` с `BatchNorm`, `Adam(lr=1e-3)` и `EarlyStopping`. Он даёт лучший validation-score в части A и сохраняет корректный протокол выбора модели, где test используется только один раз в конце. Если развивать решение дальше, я бы попробовал `scheduler` для `Adam/SGD` и сравнение `AdamW` или более аккуратный подбор `weight_decay` на том же split.
