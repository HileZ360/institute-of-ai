# HW08-09 - PyTorch MLP: регуляризация и оптимизация обучения

## 1. Кратко: что сделано

В качестве основного датасета выбран `EMNIST(split="balanced")`. В сравнении с классическим MNIST, он представляет более сложную задачу классификации, поскольку содержит 47 классов, что делает сравнение конфигураций MLP более показательным.

В части A выполнены четыре эксперимента: базовая модель E1, модель с Dropout (E2), модель с BatchNorm (E3), обучение лучшей конфигурации с использованием EarlyStopping (E4).

В части B на той же архитектуре проведены три обязательных эксперимента: O1 с заведомо слишком большим `lr`, O2 с заведомо слишком маленьким `lr` и O3 с оптимизатором `SGD+momentum` и `weight_decay`.

## 2. Среда и воспроизводимость

Входные для воспроизводимости: Python 3.13.5, `torch` 2.10.0, `torchvision` 0.25.0. Запуск зафиксирован на `cpu`, что обеспечивает воспроизводимость результатов. При необходимости устройство можно переопределить через `HW0809_DEVICE=auto/cuda/mps`. Во всех экспериментах использовался `seed=42`.

Для воспроизведения результатов достаточно открыть `homeworks/HW08-09/HW08-09.ipynb` из корня репозитория и выполнить ноутбук целиком. При необходимости путь к локально сохраненному EMNIST можно задать через переменную `HW0809_DATA_ROOT`.

## 3. Данные

В работе использован датасет `EMNIST(split="balanced")`. Разбиение данных имеет вид `train/val/test = 90240 / 22560 / 18800`. В качестве преобразований применялись `ToTensor()` и `Normalize((0.5,), (0.5,))`.

Данный набор содержит 47 классов рукописных символов, поэтому задача требует аккуратного выбора регуляризации и устойчивой оптимизации. Размер входа остается стандартным (`1 x 28 x 28`).

## 4. Базовая модель и обучение

Базовая архитектура MLP имеет вид: `Flatten -> Linear(784, 512) -> ReLU -> Linear(512, 256) -> ReLU -> Linear(256, 47)`.

В качестве функции потерь использовалась `CrossEntropyLoss`. Базовым оптимизатором для части A выбран `Adam(lr=1e-3)`. Размер батча составил 128. Максимальное число эпох задавалось следующим образом: 10 для E1-E3, 15 для E4, 6 для O1-O2 и 10 для O3. Для ранней остановки использовался `EarlyStopping` с `patience=3` по метрике `val_accuracy`.

## 5. Часть A (S08): регуляризация (E1-E4)

Эксперимент E1 соответствует базовой конфигурации с двумя скрытыми слоями `[512, 256]`. Не использовались Dropout и без BatchNorm.

Эксперимент E2 повторяет E1. После скрытых слоев добавлен `Dropout(p=0.3)`.

Эксперимент E3 также основан на E1. После каждого `Linear` добавлен слой `BatchNorm1d`.

Эксперимент E4 представляет собой обучение лучшей конфигурации из E2 и E3 с применением `EarlyStopping`.

## 6. Часть B (S09): LR, оптимизаторы, weight decay (O1-O3)

В эксперименте O1 использован заведомо слишком большой шаг обучения: `Adam(lr=1.0)`.

В эксперименте O2, использован заведомо слишком маленький шаг обучения: `Adam(lr=1e-5)`.

В эксперименте O3 применялся оптимизатор `SGD(lr=5e-2, momentum=0.9, weight_decay=1e-4)`.

## 7. Результаты

Связанные артефакты сохранены в репозитории:

- таблица результатов: `./artifacts/runs.csv`
- лучшая модель: `./artifacts/best_model.pt`
- конфигурация лучшей модели: `./artifacts/best_config.json`
- кривые лучшего эксперимента: `./artifacts/figures/curves_best.png`
- кривые для крайних значений learning rate: `./artifacts/figures/curves_lr_extremes.png`

По результатам части A лучшим оказался эксперимент E4, построенный на конфигурации E3 с BatchNorm. Наибольшее значение `val_accuracy` составило 0.8561, а итоговая `test_accuracy` для финальной модели - 0.8521.

Базовый эксперимент E1 показал `best_val_accuracy=0.8434`. Добавление Dropout в E2 не улучшило результат: `best_val_accuracy=0.8352`. Использование BatchNorm в E3 привело к росту качества до `0.8551`.

В части B эксперимент O1 с `lr=1.0` практически сорвал обучение: `best_val_accuracy=0.0227`, `best_val_loss=3.9718`. Эксперимент O2 с `lr=1e-5` продемонстрировал слишком медленную сходимость: `best_val_accuracy=0.7004`, `best_val_loss=1.3230`. Эксперимент O3 с `SGD+momentum+weight_decay` дал конкурентный результат (`best_val_accuracy=0.8517`, `best_val_loss=0.4463`). Итоговая модель, в соответствии с заданием, выбиралась только по части A.

## 8. Анализ

На датасете EMNIST Balanced даже базовая MLP-модель демонстрирует устойчивое обучение. Различия между конфигурациями становятся отчетливо заметны по validation-метрикам. Эксперимент E1 дает приемлемое качество, но уступает конфигурации с нормализацией. Вариант E2 с Dropout не привел к улучшению – значение `val_accuracy` - ниже, чем у базовой модели.

Эксперимент E3 показал, что BatchNorm в данной задаче работает эффективнее. Нормализация стабилизировала обучение и обеспечила лучший результат среди моделей-кандидатов части A до применения `EarlyStopping`.

В E4 к той же конфигурации применили `EarlyStopping`. Ранняя остановка сработала после 15-й эпохи. Лучшими оказались веса, сохраненные на 12-й эпохе. Получили прирост по `val_accuracy` относительно E3. После выбора модели она была один раз оценена на тестовой выборке и показала `test_accuracy=0.8521`.

Эксперименты части B демонстрируют влияние параметров оптимизации на динамику обучения. В O1 слишком большой learning rate портит процесс оптимизации. Значение loss остается около 4, а accuracy удерживается на уровне, близком к случайному угадыванию. В O2 слишком маленький learning rate приводит к обратной ситуации: обучение остается стабильным, но слишком медленным. За отведенное число эпох не выдает сильного результата.

Эксперимент O3 отражает более практический сценарий настройки: `SGD+momentum` с `weight_decay` обучается стабильно и дает сильный результат, особенно по `val_loss`. Поскольку правило выбора финальной конфигурации в данной работе закреплено в части A, он не должен подменять итоговую модель. Для выбранного датасета оптимальным итоговым решением является MLP [512, 256] с BatchNorm, Adam(lr=1e-3) и EarlyStopping.

## 9. Итоговый вывод

В качестве итоговой конфигурации целесообразно выбрать E4: `MLP [512, 256]` с BatchNorm, `Adam(lr=1e-3)` и `EarlyStopping`. Этот вариант показывает наилучший результат по validation-метрике в части A и сохраняет корректный протокол выбора модели, при котором тестовая выборка используется только один раз для финальной оценки.
