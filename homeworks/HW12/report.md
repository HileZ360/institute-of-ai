# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv`; прогнозировалось следующее почасовое значение колонки `target`.
- Разбиение организовано по времени: `train / validation / test = 70% / 15% / 15%`.
- Сравнивались четыре обязательных подхода: `B1` (`naive-last`), `B2` (`moving-average`), `B3` (`Ridge` на лаговых, rolling- и календарных признаках), `R1` (`GRU` на оконном представлении ряда).
- Лучшим оказался `B3`: он дал минимальный `validation MAE`, а затем был один раз проверен на `test` по корректному протоколу.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: хронологический split без перемешивания
- Какие доли или интервалы использовались:
  - `train`: `3024` точки, `2025-01-01 00:00:00` .. `2025-05-06 23:00:00`
  - `validation`: `648` точек, `2025-05-07 00:00:00` .. `2025-06-02 23:00:00`
  - `test`: `648` точек, `2025-06-03 00:00:00` .. `2025-06-29 23:00:00`
 `random split` здесь некорректен: он смешивает прошлые и будущие точки между train/validation/test, поэтому модель получает утечку из будущего и метрики становятся нереалистично оптимистичными.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: обязательные `lag_1`, `lag_7`, `lag_14` и дополнительно `lag_24`, `lag_168`
- Какие rolling-признаки использовались: `rolling_mean_7`, `rolling_std_7`, `rolling_mean_24`, `rolling_std_24`
- Какие календарные признаки использовались: `day_of_week`, `hour`, `is_weekend`
- Как обрабатывались пропуски после построения лагов/rolling: строки в начале ряда, для которых не хватает истории, удалялись
- Использовалось ли масштабирование и где именно оно обучалось:
  - для `B3`: `StandardScaler`, `fit` только на `train`
  - для `R1`: `StandardScaler` по таргету, `fit` только на `train`

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (`naive-last`): прогноз равен последнему наблюдаемому значению ряда
- B2 (`moving-average`): прогноз как скользящее среднее по окну `2`
- B3 (`ridge-lag-features`): `Ridge(alpha=1.0)` на лаговых, rolling- и календарных признаках
- R1 (`gru-forecast`): однослойная `GRU(hidden_size=32)` на окнах длиной `168`

Дополнительно:

- Основная метрика выбора лучшей модели: `validation MAE`
- Дополнительные метрики: `RMSE`, `MAPE`
- `window_size` для `GRU`: `168`
- Batch size: `64`
- Максимальное число эпох: `25`
- Optimizer: `Adam`
- Learning rate: `1e-3`
- Как сохранялась лучшая модель: по минимуму `validation MAE`, сохранялся `state_dict()` в `./artifacts/best_gru.pt`

## 7. Анализ

B1 сработал нормально из-за сильной автокорреляции, соседние значения похожи. B2 не помог: сглаживание теряет локальную динамику. Лучший результат у B3: лаги (24, 168) + календарные признаки дали модели понимание суточной и недельной структуры.
Ridge оказался простым и эффективным.
GRU не оправдал сложность — хуже даже простых подходов.
Критично: без random split, сдвиги в rolling-фичах и корректный scaling. Test использовался только один раз — для финальной проверки. Ошибки в основном на резких скачках, но в целом модель хорошо держит уровень и сезонность.
## 8. Итоговый вывод

Лучшее решение это Ridge на лагах (24, 168) и календарных признаках. Даёт стабильный результат, легко обучается и интерпретируется. GRU здесь избыточна и не даёт прироста.
